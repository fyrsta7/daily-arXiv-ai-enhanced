### [69] [Misleading Microbenchmarks on the Java Virtual Machines](http://arxiv.org/abs/2605.23570v1)

Selection reason: cs.SE 论文按全量策略收录。

TL;DR: 微基准测试在JVM上可能因收集不真实剖析信息而产生误导性结果，本文展示了该问题并扩展了指南以提高结果代表性。


<details>
  <summary>Details</summary>

Result: 即使遵循现有指南，微基准测试在特定条件下仍产生误导性结果。

Conclusion: 需要扩展指南以考虑剖析真实性，使微基准测试结果更贴近真实应用场景。

中文摘要：开发者经常使用微基准测试来选择方法或类的最优实现。在Java虚拟机（JVM）上，这通常使用Java微基准测试框架（JMH）来完成，该框架解决了在JVM上测量代码性能的常见陷阱。然而，即使遵循JMH指南也无法克服上下文的基本问题。微基准测试本质上是在隔离环境中执行代码，不会受到其他应用程序代码竞争CPU资源（如缓存或分支预测器容量）的干扰。在具有分层动态编译的托管运行时（如JVM）上，编译决策的投机性和基于剖析的性质意味着代码性能高度依赖于早期执行期间收集的剖析信息。由于剖析信息通常还包括分支概率和接收者类型（除了代码热度指标），设计不良的微基准测试可能导致JVM收集不真实的剖析信息，从而产生激进但具有误导性的优化，这在真实应用程序中不会发生。在本文中，我们展示了在诱导JVM收集不真实剖析信息的条件下使用微基准测试，尽管遵循现有指南，仍会产生误导性结果。我们还扩展了这些指南，建议采取行动使微基准测试结果更具代表性。

Original abstract: Developers often use microbenchmarks to choose the most performant implementation of a method or a class. On the Java Virtual Machine (JVM), this is commonly done using the Java Microbenchmark Harness (JMH) which addresses common pitfalls of measuring code performance on the JVM. However, even using JMH guidelines cannot overcome the fundamental issue of context.
  Microbenchmarks inherently execute code in isolation, without interference from other application code competing for CPU resources, such as cache or branch-predictor capacity. On managed runtimes with tiered dynamic compilation, such as the JVM, the speculative, profile-driven nature of compilation decisions means that code performance is highly dependent on profiles collected during early execution. Because profiles usually include also branch probabilities and receiver types (besides code hotness metrics), a badly designed microbenchmark may cause the JVM to collect an unrealistic profile, resulting in aggressive, yet misleading, optimizations, that would not occur in a real application.
  In this paper, we demonstrate how using microbenchmarks under conditions that induce the JVM to collect unrealistic profiles yields misleading results despite following existing guidelines. We also extend these guidelines by suggesting actions to make the microbenchmark results more representative.
